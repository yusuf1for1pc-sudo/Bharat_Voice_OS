Step 1 — Create the Accessibility Service in Android native code

Flutter does not directly support Accessibility Services. You write this in Kotlin inside your Flutter project's Android folder.

Create android/app/src/main/kotlin/com/bharatvoiceos/BharatAccessibilityService.kt:

kotlin
import android.accessibilityservice.AccessibilityService
import android.view.accessibility.AccessibilityEvent
import android.view.accessibility.AccessibilityNodeInfo

class BharatAccessibilityService : AccessibilityService() {

    override fun onAccessibilityEvent(event: AccessibilityEvent?) {
        // This fires every time anything changes on screen
        // We use it to know when Ola has loaded and is ready
    }

    override fun onInterrupt() {}

    // The main function — takes an action command and executes it
    fun executeAction(action: String, parameters: Map<String, String>) {
        when(action) {
            "book_ola_cab" -> bookOlaCab(parameters["destination"] ?: "")
            "send_whatsapp" -> sendWhatsapp(parameters["contact"] ?: "", parameters["message"] ?: "")
        }
    }

    private fun bookOlaCab(destination: String) {
        // Open Ola
        val intent = packageManager.getLaunchIntentForPackage("com.olacabs.customer")
        intent?.let { startActivity(it) }
        
        // Wait for Ola to load then find the destination field
        // Use a Handler to wait 2 seconds for the app to open
        android.os.Handler(mainLooper).postDelayed({
            val rootNode = rootInActiveWindow ?: return@postDelayed
            
            // Find destination input by resource ID
            val nodes = rootNode.findAccessibilityNodeInfosByViewId(
                "com.olacabs.customer:id/et_pickup_drop"
            )
            
            if (nodes.isNotEmpty()) {
                val destinationField = nodes[0]
                // Tap the field
                destinationField.performAction(AccessibilityNodeInfo.ACTION_CLICK)
                // Type the destination
                val args = android.os.Bundle()
                args.putCharSequence(
                    AccessibilityNodeInfo.ACTION_ARGUMENT_SET_TEXT_CHARSEQUENCE,
                    destination
                )
                destinationField.performAction(AccessibilityNodeInfo.ACTION_SET_TEXT, args)
            }
        }, 2000)
    }
}

How It Actually Works — The Vision Agent Loop
This is the core technical loop that makes general agent mode possible.

Step 1 — Task comes in You speak: "Swiggy pe biryani order karo, sabse sasta wala, Marine Drive deliver karo" Saarika transcribes. Sarvam-M extracts: task = order food, app = Swiggy, item = biryani, preference = cheapest, address = Marine Drive.

Step 2 — App opens Accessibility Service or a simple intent launches Swiggy. Android brings it to the foreground.

Step 3 — Screenshot taken The agent takes a screenshot of the current screen. This is completely free on Android — one line of code using MediaProjection API.

Step 4 — Vision model analyses the screenshot Screenshot is sent to a vision-capable LLM. You send it with a prompt like: "This is a screenshot of Swiggy. The task is to order the cheapest biryani and deliver to Marine Drive. What is the single next action to take? Respond with: tap at coordinates X,Y or type text T in the currently focused field or scroll direction."

The model looks at the screen and says: "Tap at coordinates 540, 320" — which happens to be the search bar.

Step 5 — Action executed Accessibility Service taps those coordinates. Or types the text. Or scrolls.

Step 6 — Screenshot again New screenshot of the updated screen. Back to Step 4. Loop repeats until task is complete or agent says done.

This is called a vision-action loop and it is exactly how Perplexity's browser agent, Anthropic's Computer Use, Google's Project Mariner, and Apple Intelligence all work. The underlying principle is identical across all of them.

The Honest Challenges of General Agent Mode
Speed — Each loop iteration takes 2 to 4 seconds. Booking an Ola cab might take 8 to 10 loop iterations meaning 20 to 40 seconds total. That is acceptable for real use but slower than the hardcoded demo flow which runs in 5 seconds. For demo day use hardcoded. For real product use vision agent.

Cost per task — Each screenshot sent to a vision model costs money. A 10-step task costs 10 vision API calls. At Gemini free tier (15 requests/minute, 1500 requests/day) this is fine for demo and early real use. At scale you need to budget for this.

Hallucination — The vision model might misidentify what is on screen and tap the wrong thing. You handle this with the max_steps safety limit and the confirmation gate before any payment or send action. The model can also get stuck in a loop tapping the same thing repeatedly — the history prevents this but you still need the step limit.

Let me break down exactly what the challenge is and how to handle it.

The Actual Problem — It Is Three Separate Problems
Problem 1 — Speech sounds similar but written differently

Hindi and Urdu are phonetically almost identical. "Main ghar ja raha hoon" sounds the same whether the speaker is a Hindi speaker or an Urdu speaker. But Hindi is written in Devanagari and Urdu in Nastaliq/Arabic script. If the system detects "Hindi" and transcribes in Devanagari, an Urdu speaker gets output in a script they may not read.

Same issue exists between Maithili and Hindi, Bhojpuri and Hindi, Awadhi and Hindi. All sound similar. All have distinct written forms.

Problem 2 — Input language vs output language vs script preference

Someone speaks in Hinglish — "Mujhe ek cab book karni hai to Marine Drive please." What language is this? Technically it is Hindi grammar with English vocabulary written in Latin script by most people when typing. But Saarika might transcribe it in Devanagari. The user might want the response in English because that is what they are comfortable reading. Or in Hindi Devanagari. Or in Hinglish Latin script.

Input language, output language, and script are three separate variables that can all differ for the same user.

Problem 3 — Same words mean different things in different languages

"Kal" means tomorrow in Hindi and yesterday in Tamil. "Anna" means elder brother in Tamil and is a common name in North India. If the system misidentifies the language, the meaning of the entire sentence changes.

How Sarvam Actually Handles This
Sarvam Saarika v2.5 does automatic language identification but with important nuances. It identifies based on acoustic patterns and vocabulary together. Hindi and Urdu speech are identified separately with reasonable accuracy because while they sound similar, vocabulary and pronunciation patterns diverge enough for a well-trained model to distinguish them in most cases.

However it is not perfect. Edge cases exist. A highly educated Urdu speaker who uses formal Urdu vocabulary will be identified more accurately than a casual speaker who uses primarily common words shared between Hindi and Urdu.

The returned language_code from Saarika tells you what it detected. You can use this to make downstream decisions.

def detect_script(text):
    devanagari_chars = sum(1 for c in text if '\u0900' <= c <= '\u097F')
    latin_chars = sum(1 for c in text if c.isascii() and c.isalpha())
    arabic_chars = sum(1 for c in text if '\u0600' <= c <= '\u06FF')
    
    total = devanagari_chars + latin_chars + arabic_chars
    if total == 0:
        return "unknown"
    
    if latin_chars / total > 0.7:
        return "latin"  # Hinglish typed, English typed
    elif devanagari_chars / total > 0.7:
        return "devanagari"  # Hindi, Marathi, Nepali
    elif arabic_chars / total > 0.7:
        return "arabic"  # Urdu
    else:
        return "mixed"  # Code-mixed

            glowWidth.dpToPx(),
            (screenHeight * 0.6f).toInt() // 60% of screen height
        )
        
        // Right edge  
        val rightView = createEdgeView(
            Gravity.END or Gravity.CENTER_VERTICAL,
            glowWidth.dpToPx(),
            (screenHeight * 0.6f).toInt()
        )
        
        // Apply gradient — brighter in center, fades toward top and bottom
        val gradient = GradientDrawable(
            GradientDrawable.Orientation.TOP_BOTTOM,
            intArrayOf(
                Color.TRANSPARENT,
                color,
                color,
                Color.TRANSPARENT
            )
        )
        
        leftView.background = gradient
        rightView.background = gradient
        
        // Start pulse animation
        startPulseAnimation(leftView, rightView)
    }
    
    private fun startPulseAnimation(vararg views: View) {
        val animator = ValueAnimator.ofFloat(0.3f, 1.0f, 0.3f).apply {
            duration = 2000
            repeatCount = ValueAnimator.INFINITE
            repeatMode = ValueAnimator.RESTART
            interpolator = AccelerateDecelerateInterpolator()
            
            addUpdateListener { animation ->
                val alpha = animation.animatedValue as Float
                views.forEach { it.alpha = alpha }
            }
        }
        animator.start()
    }
    
    fun showActionGlow() {
        // Faster brighter pulse when actively tapping
        // Alpha goes 0.6 to 1.0 at 800ms instead of 2000ms
    }
    
    fun showSuccessGlow() {
        // Switch to green, pulse once, then fade out over 1.5 seconds
        val fadeOut = AlphaAnimation(1.0f, 0.0f).apply {
            duration = 1500
            startOffset = 500
            fillAfter = true
        }
    }
    
    fun hide() {
        // Remove overlay views from window manager
    }
}
The key flag is FLAG_NOT_TOUCHABLE — this means the glow overlay exists visually but all touches pass straight through to whatever app is underneath. The user can still tap Ola normally. They do not accidentally tap the glow.
